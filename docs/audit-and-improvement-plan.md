# Полный аудит проекта «Генератор расклеек СПН»

Дата аудита: 17.06.2026
Репозиторий: `deputat36/etagi`

## 1. Назначение проекта

Проект — статический генератор печатных макетов для специалистов по недвижимости. Его основная задача: дать СПН возможность быстро собрать объявление под расклейку, листовку или простую печатную рекламу без дизайнера и сложных редакторов.

Текущая цель сформулирована правильно: СПН должен выбрать задачу, выбрать шаблон, заменить данные, добавить фото/QR при необходимости, проверить качество и отправить в печать/PDF.

## 2. Текущее состояние

### Что уже хорошо

1. Проект статический и подходит для GitHub Pages.
2. Основной сценарий понятный: задача → шаблон → данные → печать.
3. Есть модульная структура JS: состояние, шаблоны, рендер, проверка качества, сохранение.
4. Есть JSON-библиотека шаблонов.
5. Есть предпросмотр А4.
6. Есть печатные стили.
7. Есть сохранение макета в браузере и экспорт/импорт JSON.
8. Есть отдельный профиль СПН: имя и телефон можно сохранять отдельно от макета.
9. Добавлен блок «Состав макета»: можно включать и выключать части объявления.
10. Район теперь вводится вручную, без навязанных пресетов.

### Главные риски текущей версии

1. QR пока не настоящий: отображается заглушка `QR`, а не реальный код.
2. Адаптивная перестройка блоков пока базовая: блоки скрываются, но нет умного перераспределения размеров текста, фото и контактного блока.
3. Проверка качества есть, но она пока не даёт достаточно точных рекомендаций по конкретному типу макета.
4. Шаблоны хранятся в одном большом `data/templates.json`. По мере роста библиотеки файл станет сложнее поддерживать.
5. Стили сильно сжаты в одну строку. Это ухудшает поддержку и увеличивает риск ошибок при ручной доработке.
6. Нет полноценного режима «новичок / опытный».
7. Нет системы пользовательских блоков: сейчас можно только включать/выключать заранее заданные блоки, но нельзя добавить дополнительный произвольный блок.
8. Нет drag-and-drop порядка блоков.
9. Нет реального экспорта в PNG/JPG. Есть печать/PDF через браузер.
10. Нет автотестов и автоматической проверки JSON/JS перед публикацией.

## 3. Аудит архитектуры

### HTML

`index.html` выполняет роль всей оболочки приложения. Это нормально для текущего размера проекта. Но при росте интерфейса станет сложно поддерживать один большой файл.

Сильные стороны:

- понятная последовательность экранов;
- все элементы доступны без сборки;
- подходит для GitHub Pages.

Слабые стороны:

- много логики завязано на id элементов;
- при добавлении новых блоков нужно синхронно менять HTML, JS state, JS app и render;
- нет шаблонной системы UI-компонентов.

### JavaScript

Текущая структура хорошая для статического проекта:

- `state.js` — справочники и начальное состояние;
- `templates.js` — загрузка и фильтрация шаблонов;
- `app.js` — управление интерфейсом;
- `render.js` — вывод макета;
- `quality.js` — проверка качества;
- `storage.js` — localStorage;
- `utils.js` — утилиты.

Главная проблема: `app.js` становится центральным «комбайном». В нём уже есть работа с формой, шаблонами, профилем, блоками, сохранением, печатью, проверкой качества и статусами. Его нужно постепенно разделить.

Рекомендуемое разделение:

- `profile.js` — профиль СПН;
- `block-controls.js` — управление блоками;
- `form.js` — чтение/синхронизация формы;
- `print.js` — печать и предпросмотр;
- `presets.js` — быстрые пресеты;
- `app.js` — только инициализация и связывание модулей.

### CSS

Сейчас `app.css` содержит основную визуальную систему, но файл сжат в одну строку. Это плохо для сопровождения.

Нужно:

1. Отформатировать CSS.
2. Разделить стили на логические файлы:
   - `base.css`;
   - `layout.css`;
   - `controls.css`;
   - `flyer.css`;
   - `print.css`;
   - `responsive.css`.
3. Оставить единый импорт или подключить несколько файлов напрямую.

### JSON-шаблоны

Шаблоны — сильная сторона проекта. Но один большой файл со временем станет неудобным.

Нужно разделить:

- `data/templates/seller.json`;
- `data/templates/buyer.json`;
- `data/templates/object.json`;
- `data/templates/newbuild.json`;
- `data/templates/service.json`;
- `data/templates/rent.json`;
- `data/templates/brand.json`;
- `data/templates/private.json`.

И обновить загрузчик шаблонов, чтобы он собирал массив из нескольких файлов.

## 4. Аудит пользовательского сценария

### Текущий сценарий

1. СПН открывает сайт.
2. Выбирает задачу.
3. Выбирает шаблон.
4. Вводит данные.
5. Выбирает состав макета.
6. Проверяет качество.
7. Печатает.

Сценарий правильный, но его нужно сделать более «безошибочным».

### Что мешает удобству

1. Много полей на одном экране.
2. Пользователь может не понимать, что лучше отключить, если макет переполнен.
3. Не хватает «умных режимов»:
   - макет для подъезда;
   - макет для витрины;
   - макет для срочной продажи;
   - макет без бренда;
   - макет с крупным телефоном;
   - макет с фото объекта;
   - макет только с QR.
4. Нет готовых связок «тип задачи → оптимальный состав блоков».

### Как улучшить

Добавить режимы в один клик:

- «Максимально читаемо» — крупный заголовок, крупный телефон, мало текста.
- «Экономно на 4» — 4 объявления на А4, без фото, короткий текст.
- «С фото объекта» — 1 или 2 объявления на А4, фото включено.
- «Без бренда» — частный стиль, без логотипа.
- «Для подъезда» — отрывные телефоны, минимум текста.
- «Для витрины» — 1 крупный макет, фото, QR, без отрывных телефонов.

## 5. Аудит адаптивности макета

Сейчас адаптивность реализована частично: блоки можно скрывать, а HTML просто не выводит ненужные секции. Это правильная база.

Следующий уровень — не просто скрывать блоки, а менять поведение оставшихся:

- если фото выключено — заголовок и контактный блок должны стать крупнее;
- если описание выключено — преимущества могут стать крупнее;
- если контактный блок выключен — QR/фото/описание должны занимать свободное место;
- если выбрано 4 на А4 — система должна автоматически предлагать скрыть фото и описание;
- если выбран 1 макет на А4 — система может предлагать включить фото и QR.

Для этого нужен новый слой: `layoutRules.js`.

Пример логики:

- `printCount >= 4` → плотный режим, максимум 3 преимущества, маленькое фото или без фото.
- `photoMode !== none && printCount === 1` → фото крупнее.
- `showDescription === false` → headlineScale +0.1, phoneScale +0.1.
- `showContact === false` → отключить отрывные телефоны.

## 6. Аудит качества печати

Печать работает через браузер. Это простой и правильный вариант для v1.

Проблемы:

1. Разные браузеры могут печатать чуть по-разному.
2. Нужно явно подсказать включать фоновые изображения.
3. Нет тестовой страницы «проверить печать».
4. Нет визуальных линий реза/разделения.
5. Нет безопасных полей под принтер.

Нужно добавить:

- режим «Показать линии реза»;
- режим «Без полей / с безопасными полями»;
- тестовый шаблон печати;
- инструкцию «как печатать в Chrome/Edge»;
- кнопку «перед печатью проверить масштаб».

## 7. Аудит визуального дизайна

Текущий дизайн интерфейса нормальный для рабочего инструмента. Но сами макеты пока выглядят скорее как технические шаблоны, а не как дизайнерская система.

Нужно разработать 4 визуальных стиля:

1. «Этажи классика» — красный фирменный, логотип, понятная структура.
2. «Экономная расклейка» — минимум заливок, меньше расход краски.
3. «Частное объявление» — без явной фирменности, нейтрально.
4. «Премиум / объект с фото» — для витрины, презентации, дорогих объектов.

Каждый стиль должен быть не просто цветом, а набором правил:

- размер заголовка;
- размер телефона;
- форма контактного блока;
- наличие рамки;
- плотность;
- расположение фото;
- наличие/отсутствие отрывных телефонов.

## 10. Аудит безопасности и устойчивости

Проект не работает с сервером, поэтому риски низкие.

Но есть технические моменты:

- пользовательские тексты должны экранироваться при выводе;
- импорт JSON должен проверяться на структуру;
- большие фото могут тормозить браузер;
- localStorage может переполниться, если сохранять фото в base64.

Сейчас фото в автосохранении очищаются — это правильно. Но нужно дополнительно:

- ограничить размер фото;
- сжимать фото перед вставкой;
- показывать предупреждение, если фото слишком большое;
- добавить безопасную нормализацию импортированного JSON.

## 13. Итог

Проект уже имеет правильную основу: статический сайт, понятный сценарий, шаблоны, печать, профиль СПН и управляемый состав макета. Но пока это не полноценный конструктор, а гибкий генератор шаблонов.

Чтобы довести его до рабочего офисного инструмента, нужно двигаться в таком порядке:

1. стабилизировать текущую версию;
2. добавить умную адаптацию блоков;
3. сделать настоящий QR;
4. улучшить печать;
5. расширить библиотеку шаблонов;
6. после тестирования с СПН — упростить интерфейс и добавить пользовательские шаблоны.

# Расширение DOM-проверок

Документ фиксирует техническую задачу для `tools/validate-assets.mjs`.

## Текущее состояние

`validate:assets` уже проверяет:

- ссылки на локальные файлы из HTML;
- подключение JSON-пакетов шаблонов из `assets/js/templates.js`;
- импорты JS-модулей;
- дубли локальных script/stylesheet;
- id из массивов `fields`, `checks` и `blockVisibility` в `assets/js/app.js`;
- прямые обращения `$('<id>')` из `assets/js/app.js`;
- динамические id, которые создаются в JS, а не лежат в исходном HTML.

## Что нужно расширить

При следующем безопасном патче нужно заменить узкую функцию извлечения `$('<id>')` на общую функцию прямых DOM-id.

Она должна учитывать:

```text
$('<id>')
document.getElementById('<id>')
document.querySelector('#<id>')
document.querySelectorAll('#<id>')
```

## Зачем это нужно

Если в `assets/js/app.js` появится прямое обращение к DOM через стандартный API браузера, валидатор должен поймать ситуацию, когда такого `id` нет в `index.html` и он не описан как динамический.

## Безопасное поведение

Проверка должна:

- не требовать динамические элементы в исходном HTML;
- оставлять список `dynamicAppIds` единственным местом для исключений;
- не анализировать сложные CSS-селекторы;
- проверять только простые обращения вида `#id`;
- не менять поведение интерфейса.

## Рекомендуемая реализация

В `tools/validate-assets.mjs`:

1. Переименовать `extractDollarLiteralIds` в `extractDirectAppDomIds`.
2. Добавить несколько регулярных выражений для простых DOM-доступов.
3. Возвращать уникальный список id.
4. Оставить существующую проверку этих id по `index.html` и `dynamicAppIds`.

## После реализации

Запустить:

```bash
npm run validate:assets
npm run validate
```

Если проверка выполняется через GitHub connector и локально команды не запускались, это нужно честно указать в отчёте.

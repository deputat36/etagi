# Полный аудит и самостоятельный план развития генератора расклеек

Дата аудита: 2026-07-09

## 1. Короткий вывод

Генератор расклеек уже вырос из простой страницы в полноценный рабочий инструмент для СПН: есть библиотека шаблонов, печатный предпросмотр А4, QR, фото, отрывные телефоны, контроль качества, режимы интерфейса, режим `Новичок`, Wizard Flow, сохранение макетов, задание на расклейку, отчёт, история отчётов, CSV-экспорт, документация и много автоматических проверок.

Главная проблема проекта теперь не в нехватке функций, а в упаковке функций. В интерфейсе уже есть почти всё, но новичку сложно понять, что делать сначала, что обязательно, что можно не трогать, какой шаблон безопасен, когда нужен менеджер и почему печать может быть заблокирована.

Главный принцип дальнейшей работы:

```text
Не добавлять хаотично новые кнопки.
Собирать существующие функции в понятный рабочий сценарий.
```

## 2. Текущее состояние проекта

### 2.1. Что уже хорошо

1. Проект работает как статический инструмент на GitHub Pages без серверной зависимости.
2. Основной путь уже понятен на уровне продукта: задача → шаблон → данные → проверка → печать → задание → отчёт.
3. Шаблоны лежат в отдельных JSON-пакетах, поэтому библиотеку можно расширять без переписывания ядра.
4. Есть локальные пакеты под Борисоглебск, ЖК «Теллерманов сад», подъездную расклейку, A/B-тесты и доверительные сценарии.
5. Поиск шаблонов индексирует не только название и теги, но и тексты макета, дополнительные поля, подписи и брендовые строки.
6. Есть режимы интерфейса: `Новичок`, `Быстро`, `Расширенно`.
7. Режим `Новичок` уже защищает от рискованных шаблонов, помогает с печатью и не должен перехватывать весь интерфейс глобальными обработчиками.
8. Wizard Flow уже появился и ведёт пользователя по шагам.
9. Количество объявлений на А4 уже продублировано в начале мастера, а вариант `2 на А4` выделен как частый.
10. Контроль качества сильный: проверяет телефон, читаемость, QR, фото, перегрузку текста, плотную печать, безопасные поля, отрывные телефоны и фирменность.
11. Есть задание на расклейку и отчёт после расклейки, то есть инструмент закрывает не только печать, но и полевой цикл.
12. Есть история отчётов, повтор прошлой расклейки и CSV-экспорт.
13. В проекте сильная система валидации через `npm run validate` и набор `validate:*` проверок.

### 2.2. Главная зона риска

Интерфейс слишком насыщен. В одном сайдбаре одновременно находятся:

- выбор цели;
- поиск и фильтры шаблонов;
- редактирование всех данных;
- быстрые режимы;
- состав макета;
- порядок блоков;
- фото и QR;
- печать;
- расширенные настройки;
- контроль качества;
- сохранение;
- помощь;
- helper-блоки;
- задание на расклейку;
- отчёт после расклейки.

Для опытного пользователя это гибко. Для новичка это выглядит как сложная CRM или дизайнерский редактор, хотя задача должна ощущаться проще: выбрать ситуацию, заполнить минимум данных, проверить и распечатать.

## 3. Аудит по направлениям

### 3.1. Пользовательский сценарий СПН

Сильная сторона: путь от макета до отчёта уже существует.

Проблема: сценарий пока больше собран технически, чем методически. Пользователь видит много блоков до того, как понял, какой результат ему нужен.

Что улучшить:

1. Сделать Wizard Flow главным ощущением интерфейса для новичка.
2. В первом шаге оставить только цель, формат А4 и понятную подсказку.
3. Во втором шаге показать крупные категории заготовок, а не мелкие фильтры.
4. В третьем шаге показывать карточки шаблонов с понятными метками: `новичку`, `менеджер`, `подъезд`, `с фото`, `2 на А4`.
5. В редактировании показывать сначала только поля, которые реально влияют на выбранный макет.
6. Расширенные настройки оставить, но не показывать как основной путь.

### 3.2. Режим `Новичок`

Сильная сторона: режим уже есть, фильтрует безопасные шаблоны, скрывает рискованные и защищает печать.

Проблемы:

1. Безопасность новичка сейчас основана в основном на поисковом запросе `новичку` и текстовом анализе карточек.
2. Новичку может быть непонятно, почему часть шаблонов скрыта.
3. Нет отдельной короткой подсказки на каждом шаге: что заполнить обязательно, а что можно пропустить.

Что улучшить:

1. Ввести явные признаки шаблонов: `recommended`, `level`, `scenario`, `risk`, `managerNote`.
2. Оставить поддержку тегов, но не полагаться только на текстовый поиск.
3. Добавить в Wizard Flow короткие подсказки по каждому шагу именно для новичка.
4. Сделать кнопку или блок `Показать менеджеру`, который формирует короткую сводку: шаблон, цель, телефон, качество, предупреждения.
5. Добавить предупреждение: «Этот макет требует проверки менеджера» для шаблонов с тегом `менеджер`.

### 3.3. Шаблоны

Сильная сторона: библиотека уже разбита по пакетам и расширяется безопасно.

Проблемы:

1. В некоторых сценариях шаблонов мало для реальной полевой работы.
2. Теги решают многое, но со временем могут стать хаотичными.
3. Нет единого поля, где было бы описано: кому шаблон подходит, риск, менеджерская заметка, рекомендованный формат.

Что улучшить:

1. Добавить метаданные шаблона:

```json
"office": {
  "recommended": true,
  "level": "newbie",
  "scenario": "entrance_owner",
  "risk": "low",
  "managerNote": "Можно печатать после проверки телефона"
}
```

2. Сначала внедрить это на 5–10 ключевых шаблонах, не массово.
3. Добавить валидатор для новых office-полей.
4. Расширить локальные шаблоны Борисоглебска:
   - Северный район;
   - Аэродромная;
   - центр;
   - частный сектор;
   - оценка квартиры;
   - «куплю в вашем доме»;
   - «есть покупатель по району»;
   - «после звонка соседа»;
   - «продать старую и купить новостройку».
5. По ЖК «Теллерманов сад» добавить отдельные макеты:
   - семейная ипотека;
   - маткап;
   - первый взнос;
   - студии / первое жильё;
   - 2-комнатные для семьи;
   - trade-in / продажа старого жилья;
   - ожидание старта продаж без опасных обещаний.

### 3.4. Фото и визуальная компоновка

Сильная сторона: фото уже поддерживаются, есть `photoOne`, `photoTwo`, режимы фото и CSS-раскладки.

Проблема: фото пока является одним из блоков, а не полноценным типом макета. Для продающих объектов фото должно быть композиционным центром, а не просто вставкой.

Что улучшить:

1. Ввести понятие типа компоновки:

```text
text_only
photo_top
photo_left
photo_card
newbuild_visual
agent_brand_photo
```

2. Для 1 на А4 и 2 на А4 делать фото крупным.
3. Для 4 на А4 фото разрешать только в специальных макетах.
4. Для 6–8 на А4 предупреждать, что фото почти всегда ухудшает читаемость.
5. Добавить шаблоны с фото СПН для личного бренда.
6. Добавить шаблоны с визуалом ЖК для новостроек.

### 3.5. Печать и PDF

Сильная сторона: есть печатный диалог, A4, линии реза, безопасные поля, разные количества макетов на листе, защита печати в режиме `Новичок`.

Проблемы:

1. Пользователь может не понимать, какой формат выбрать.
2. Настройки печати и количество объявлений логически относятся к началу сценария, но часть настроек всё ещё находится ближе к концу.
3. Нет явного предпросмотра «лист готов / лист рискованный» до печати.

Что улучшить:

1. Оставить выбор количества объявлений в начале мастера.
2. Добавить короткие подсказки:
   - 1 на А4 — витрина / фото / объект;
   - 2 на А4 — основной рабочий формат;
   - 4 на А4 — подъезд / экономно;
   - 6–8 на А4 — только очень короткие тексты.
3. Сделать статус печати более очевидным:
   - `Готово к печати`;
   - `Нужен телефон`;
   - `Слишком много текста`;
   - `Проверьте QR`;
   - `Нужен менеджер`.
4. Добавить отдельный чек-лист печати для пожилой аудитории: крупность телефона, контраст, короткий текст.

### 3.6. Контроль качества

Сильная сторона: это один из лучших модулей проекта. Он уже делает продукт полезнее обычного редактора макетов.

Проблемы:

1. Контроль качества в основном технический и текстовый, но не всегда объясняет бизнес-логику.
2. Не хватает проверки по цели: макет для собственника, покупателя, объекта, новостройки должен иметь разные критерии.
3. Не хватает итогового решения: печатать / показать менеджеру / сократить / выбрать другой шаблон.

Что улучшить:

1. Добавить цель-специфичные проверки:
   - для собственника: есть ли мягкий повод позвонить;
   - для покупателя: понятна ли выгода обращения;
   - для объекта: есть ли цена или причина позвонить;
   - для новостройки: нет ли опасных обещаний;
   - для частного: нет ли фирменности.
2. Добавить итоговый вердикт:

```text
Можно печатать.
Можно печатать после исправления телефона.
Лучше показать менеджеру.
Макет перегружен, выберите 2 на А4 или сократите текст.
```

3. Добавить «почему это важно» в критичных замечаниях.
4. Сохранять качество макета в отчёт после расклейки, чтобы потом сравнивать результат.

### 3.7. Задание на расклейку и отчёты

Сильная сторона: генератор уже закрывает полевой цикл, а не только печать.

Проблемы:

1. Задание и отчёт пока выглядят как дополнительные блоки, а не как продолжение бизнес-процесса.
2. Нет рейтинга связок: шаблон + район + формат + количество откликов.
3. CSV есть, но внутри интерфейса мало аналитики по истории.

Что улучшить:

1. Сделать шаги `Задание` и `Отчёт` частью Wizard Flow.
2. Добавить автоподстановку названия шаблона, района, цели и количества на А4 в отчёт.
3. В истории показывать лучшие связки:
   - больше всего звонков;
   - больше всего целевых обращений;
   - лучший район;
   - лучший шаблон;
   - слабые расклейки, которые не стоит повторять.
4. Добавить простую метку результата:

```text
повторить
изменить текст
сменить район
остановить
```

5. Добавить экспорт не только CSV, но и короткий текстовый отчёт для менеджера.

### 3.8. Документация и обучение

Сильная сторона: уже есть README, центр помощи, инструкции, чек-листы, регламент сопровождения, документация по офисным тегам и шаблонам.

Проблемы:

1. Документация местами догоняет уже сделанные функции.
2. СПН может не открыть help-страницы, если подсказка не встроена в рабочий поток.
3. Не хватает краткой методички прямо внутри мастера.

Что улучшить:

1. Добавить подсказку к каждому шагу Wizard Flow.
2. В help добавить страницу «10 готовых сценариев расклейки для Борисоглебска».
3. Добавить страницу «Ошибки расклейки, из-за которых не звонят».
4. Добавить страницу «Как выбрать 1, 2, 4, 6 или 8 макетов на А4».
5. Добавить страницу «Как менеджеру проверять макет новичка за 1 минуту».

### 3.9. Техническая архитектура

Сильная сторона: ядро разделено на модули, есть много защитных валидаторов.

Проблемы:

1. `app.js` остаётся крупным центральным файлом, который управляет слишком многими задачами.
2. Много helper-модулей вставляют DOM после загрузки страницы. Это удобно для постепенного развития, но повышает риск конфликтов.
3. Часть логики офисных сценариев основана на текстовом поиске по тегам и содержимому карточек.
4. Нет единого реестра UI-блоков и шагов Wizard Flow.

Что улучшить:

1. Постепенно вынести из `app.js` контроллеры:
   - `templateController.js`;
   - `printController.js`;
   - `layoutController.js`;
   - `qualityController.js`;
   - `savedLayoutsController.js`.
2. Не делать это одним большим коммитом. Каждый вынос должен сохранять поведение и проходить validate.
3. Добавить реестр шагов Wizard Flow и DOM-секций, чтобы helper-ы не искали элементы разрозненно.
4. Перевести часть офисных правил с анализа текста на явные данные шаблона.
5. Добавить больше тестов-валидаторов для Wizard Flow: шаги, секции, кнопки, отсутствие глобальных перехватов.

## 4. Приоритетный самостоятельный план развития

### Этап 1. Закрепить текущую стабильность

Цель: не сломать уже работающий генератор.

Задачи:

1. Обновить этот аудит и roadmap в GitHub.
2. Закрепить текущие шаги Wizard Flow в validate-проверках.
3. Добавить ручной чек-лист именно для полного сценария: новичок → макет → печать → задание → отчёт.
4. Обновить README, чтобы он отражал новый `validate:ci-config`, офисные теги и шаги Wizard Flow.
5. Проверить, что изменения режима `Новичок` не влияют на режимы `Быстро` и `Расширенно`.

### Этап 2. Сделать Wizard Flow основным путём новичка

Цель: новый СПН должен сделать первый макет без объяснений.

Задачи:

1. Разделить шаг `Цель` и шаг `Количество на А4` визуально, даже если технически они находятся в одной панели.
2. Сделать крупные карточки категорий заготовок.
3. Добавить подсказки к каждому шагу.
4. В режиме `Новичок` включать `Пошагово` по умолчанию.
5. Сделать кнопку `Далее` умнее: если не заполнен телефон или не выбран шаблон, показывать подсказку.
6. Добавить шаг `Показать менеджеру`, если шаблон имеет тег `менеджер`.

### Этап 3. Улучшить шаблоны и офисные признаки

Цель: СПН выбирает не просто шаблон, а рабочий сценарий.

Задачи:

1. Добавить office-метаданные к ключевым шаблонам.
2. Добавить валидатор office-метаданных.
3. Обновить карточки шаблонов: показывать назначение, риск, уровень, рекомендуемый формат.
4. Расширить шаблоны Борисоглебска минимум до 20 рабочих сценариев.
5. Расширить шаблоны ЖК «Теллерманов сад» по сегментам покупателей.
6. Добавить шаблоны личного бренда СПН с фото.

### Этап 4. Улучшить фото-компоновки

Цель: макеты с фото должны выглядеть как продающие объявления, а не как текст с маленькой картинкой.

Задачи:

1. Ввести `layoutType` или отдельные теги компоновки.
2. Сделать CSS-компоновки `photo_top`, `photo_left`, `photo_card`.
3. Добавить проверку качества для фото-компоновок.
4. Добавить шаблоны объектов с крупным фото.
5. Добавить шаблоны новостроек с визуалом ЖК.
6. Ограничивать фото в 6–8 макетах на А4 через предупреждения и рекомендации.

### Этап 5. Усилить контроль качества как бизнес-помощника

Цель: проверка должна не просто находить ошибки, а учить делать эффективную расклейку.

Задачи:

1. Добавить итоговый вердикт качества.
2. Добавить проверки по цели макета.
3. Добавить объяснения к критичным замечаниям.
4. Добавить кнопку `Исправить главное`.
5. Добавить менеджерскую сводку перед печатью.
6. Сохранять качество макета в отчёт после расклейки.

### Этап 6. Превратить отчёты в аналитику расклейки

Цель: понимать, какие макеты и районы дают отклики.

Задачи:

1. В истории отчётов показывать лучшие и слабые связки.
2. Добавить фильтр истории по району, шаблону и цели.
3. Добавить показатель отклика: звонки / листы и целевые обращения / листы.
4. Добавить рекомендацию по результату: повторить, изменить, остановить.
5. Добавить текстовый отчёт для менеджера.
6. В будущем — подготовить импорт в Google Таблицы или CRM, но без подключения внешних сервисов до отдельного согласования.

### Этап 7. Постепенно упростить код

Цель: проект должен оставаться поддерживаемым.

Задачи:

1. Разнести `app.js` на контроллеры маленькими PR/коммитами.
2. Добавить реестр UI-секций и шагов мастера.
3. Уменьшить количество DOM-вставок helper-ов там, где блоки стали постоянной частью продукта.
4. Оставить валидаторы как обязательный барьер перед изменениями.
5. Обновлять changelog и release note после каждого завершённого этапа.

## 5. Что делать в ближайших автономных шагах

1. Создать ручной чек-лист полного сценария: `Новичок → 2 на А4 → шаблон → телефон → проверка → печать → задание → отчёт`.
2. Обновить `docs/wizard-flow-spec.md`, чтобы там были шаги `Задание` и `Отчёт`, которые уже добавлены в код.
3. Добавить в Wizard Flow подсказку по выбору количества объявлений на А4.
4. Сделать крупные карточки категорий заготовок поверх существующих офисных фильтров.
5. Добавить в карточки шаблонов более явные метки и пояснение `почему подходит новичку / почему нужен менеджер`.
6. Добавить первый office metadata prototype для 5–10 шаблонов и валидатор.
7. Добавить менеджерскую сводку перед печатью.
8. Добавить аналитику лучших связок в истории отчётов.
9. После каждого изменения проверять, что кнопки интерфейса не зависают и печать не блокируется в обычных режимах.

## 6. Критерии успеха

Проект можно считать удобным и полезным, когда выполняются условия:

1. Новый СПН за 3–5 минут делает первый макет 2 на А4 без объяснений.
2. Телефон хорошо читается, особенно для пожилой аудитории.
3. Новичок не видит рискованные пустые и менеджерские шаблоны по умолчанию.
4. Менеджер быстро понимает, что проверять перед печатью.
5. После расклейки СПН фиксирует результат, а не просто забывает про листовки.
6. Через историю видно, какие районы, шаблоны и форматы дают обращения.
7. Опытный пользователь не теряет гибкость, но она не мешает новичку.
8. Любое изменение проходит автоматические проверки и не ломает базовый сценарий.

## 7. Правило автономной работы

Дальше можно самостоятельно выполнять безопасные улучшения в таком порядке:

1. Документация и чек-листы.
2. Валидация и страховки.
3. Небольшие UX-улучшения без изменения ядра.
4. Расширение шаблонов.
5. Улучшение Wizard Flow.
6. Улучшение отчётности.
7. Рефакторинг `app.js` маленькими частями.

Не выполнять без отдельного согласования:

- подключение внешних сервисов;
- сбор персональных данных на сервер;
- изменение фирменного стиля;
- юридически чувствительные обещания;
- массовое удаление функций;
- крупную замену архитектуры одним коммитом.

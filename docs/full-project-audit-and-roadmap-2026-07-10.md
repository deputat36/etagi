# Полный аудит и автономный план развития генератора расклеек

Дата аудита: 2026-07-10

## 1. Итоговый вывод

Проект уже является не просто генератором листовок, а полноценным локальным рабочим инструментом отдела продаж недвижимости:

```text
задача → шаблон → данные → фото / QR → проверка → печать → задание → отчёт → аналитика
```

Функционально проект развит хорошо. Главная проблема теперь не в недостатке возможностей, а в трёх вещах:

1. интерфейс и логика собраны из большого количества независимых helper-модулей;
2. новичку всё ещё нужно защищать внимание от второстепенных функций;
3. автоматические проверки в основном статические и не проходят настоящий браузерный сценарий.

Главный принцип дальнейшей работы:

```text
Не добавлять функции ради количества.
Укреплять единый рабочий сценарий СПН.
Каждое улучшение должно сокращать ошибки, время подготовки или работу менеджера.
```

## 4. Ключевые проблемы

## 4.1. P0 — runtime-архитектура

### Большое количество независимых точек запуска

`spnUiMode.js` подключает более 30 helper-модулей. Многие из них самостоятельно подписываются на `DOMContentLoaded`, создают DOM-блоки, стили и MutationObserver.

Риски:

- порядок появления блоков становится скрытой зависимостью;
- один helper может не найти блок, который другой helper создаст позднее;
- повторные observer-обновления расходуют ресурсы;
- сложнее понять, какой модуль отвечает за конкретный элемент;
- возрастает вероятность конфликтов, подобных конфликту старого Wizard Patch с новой кнопкой `Далее`.

Решение:

1. не переписывать проект одним большим коммитом;
2. ввести один runtime-bootstrap и реестр постоянных модулей;
3. постепенно убрать дубли между прямыми script-тегами и импортами;
4. для новых модулей использовать единый lifecycle-контракт;
5. ограничивать MutationObserver конкретным контейнером и debounce/RAF.

### Повторная загрузка шаблонов

`loadTemplates()` вызывается из ядра, карточек, менеджерской сводки и notice. Каждый вызов сейчас повторно загружает девять JSON-файлов с `cache: no-store`.

Последствия:

- до 36 запросов вместо 9;
- лишняя нагрузка при слабом интернете;
- разные helper-ы могут получить данные в разное время;
- интерфейс может кратковременно показывать fallback вместо office-метаданных.

Решение этапа 1: общий Promise-кеш внутри `templates.js`.

### Дублирование точек входа

Часть helper-модулей одновременно:

- подключается отдельным `<script type="module">` в `index.html`;
- импортируется через `spnUiMode.js`.

Браузер обычно исполняет одинаковый URL модуля один раз, но схема остаётся запутанной и зависит от module map и порядка загрузки.

Решение этапа 1: оставить одну точку подключения для постоянных SPN-helper-ов и добавить валидатор архитектуры.

## 4.2. P0 — отсутствие браузерного smoke-теста

Сейчас CI запускает большое количество статических проверок, но не открывает страницу в браузере.

Статический validate не обнаружит:

- зависшие кнопки;
- ошибку во время `DOMContentLoaded`;
- неверный порядок появления helper-блоков;
- бесконечный MutationObserver;
- невозможность пройти Wizard Flow;
- проблему печатной компоновки после реального рендера.

Решение:

1. сохранить текущие validate как быстрый барьер;
2. добавить браузерный smoke-тест основного сценария;
3. первый smoke должен проверить только критичный путь, без визуальных снимков;
4. после стабилизации добавить screenshot-тесты 1/2/4 на А4.

## 4.3. P1 — UX новичка

Уже выполнены автозапуск мастера, безопасные шаблоны, подсказки и мягкая кнопка `Далее`.

Остаётся:

- визуально отделить выбор цели от выбора формата А4;
- сократить количество одновременно видимых помощников;
- на каждом шаге оставить один главный призыв к действию;
- убрать устаревшие сообщения вроде «режим новичка возвращается поэтапно»;
- сделать итог шага более заметным: `готово`, `нужно заполнить`, `нужен менеджер`.

## 4.4. P1 — шаблоны и офисные данные

Office-метаданные есть только у трёх ключевых пакетов. Остальные шаблоны продолжают использовать текстовые fallback-правила.

Остаётся:

- внедрить office-метаданные в основные продажи, услуги, аренду и бренд;
- расширить пакет Борисоглебска минимум до 20 реальных сценариев;
- добавить объявления для конкретных районов, домов, наследства, срочной продажи, обмена и оценки;
- добавить брендовые шаблоны СПН с фото;
- вести статус шаблона: рабочий, тестовый, устаревший.

## 4.5. P1 — фото-компоновки

Реализовано крупное верхнее фото.

Остаётся:

```text
photo_left
photo_card
newbuild_visual
agent_brand_photo
```

Для каждого типа нужно определить:

- допустимые форматы А4;
- поведение одного и двух фото;
- положение заголовка, цены и телефона;
- поведение QR;
- правило качества;
- печатную проверку.

## 4.6. P1 — отчёты

История уже показывает рабочие и слабые связки, но отчёт сохраняет в основном заголовок, место и цифры.

Не хватает:

- `templateId`;
- цели макета;
- office-сценария;
- уровня риска;
- оценки качества на момент печати;
- режима интерфейса;
- результата менеджерской проверки;
- возможности удалить ошибочный отчёт;
- полного backup/restore локальных данных.

Без `templateId` два макета с одинаковым заголовком могут смешиваться в аналитике.

## 4.7. P1 — контроль качества

Система сильная, но распределена между `quality.js` и большим количеством helper-модулей.

Остаётся:

- единый итоговый вердикт;
- цель-специфичные проверки;
- сохранение score в отчёт;
- проверка конкретного типа фото-компоновки;
- более короткая менеджерская сводка;
- единая карта действий, а не отдельные строковые валидаторы для каждого action.

## 4.8. P2 — хранение и переносимость

Плюсы:

- фото исключаются из localStorage;
- сохраняются профиль, последний макет, именованные макеты и избранное;
- данные не отправляются наружу.

Ограничения:

- всё привязано к одному браузеру и устройству;
- очистка браузера удалит историю и макеты;
- CSV сохраняет только отчёты;
- нет единого резервного файла всего рабочего пространства.

Решение:

- добавить экспорт/импорт полного локального backup JSON;
- явно показать пользователю, что данные локальные;
- не подключать серверную синхронизацию без отдельного решения по доступу и персональным данным.

## 4.9. P2 — версия и кеширование

Текущая версия `3.85.0` уже отстаёт от объёма внесённых функций.

Проблемы:

- cache-busting есть только у одного entry-модуля;
- query-параметр родительского модуля не обновляет кеш дочерних импортов;
- много функций добавлено после release-note 3.85.0.

Решение:

1. после стабилизации выпустить 3.86.0;
2. обновить package/state/changelog/release-note;
3. внедрить единый asset-version подход;
4. не менять версию до прохождения браузерного smoke-теста.

## 4.10. P2 — поддерживаемость CSS и интерфейса

- часть CSS находится в больших однострочных файлах;
- часть стилей инжектируется из JavaScript;
- компонентные стили трудно искать и переиспользовать;
- в helper-блоках часто используется текст 9–11 px.

Решение:

- новые постоянные компоненты оформлять отдельными CSS-файлами;
- постепенно переносить крупные injected-style блоки из JS;
- добавить минимальный размер интерфейсного текста;
- проверить клавиатурную навигацию и focus-visible.

## 5. Что не нужно делать сейчас

- не переписывать проект на React/Vue/Laravel;
- не подключать сервер только ради синхронизации;
- не добавлять CRM и аналитику без модели доступа;
- не удалять быстрый и расширенный режимы;
- не менять работающий формат JSON шаблонов массово;
- не делать большой рефакторинг `app.js` одним коммитом;
- не добавлять новые helper-модули без проверки, можно ли расширить существующий.

## 6. Автономный план работы

## Этап 1. Стабилизация runtime — выполняется первым

1. Добавить Promise-кеш загрузки шаблонов.
2. Убрать дубли helper-модулей между `index.html` и `spnUiMode.js`.
3. Добавить `validate:runtime-architecture`.
4. Исправить устаревшие тексты режима.
5. Обновить README и master issue.
6. Не менять пользовательские данные и бизнес-логику.

Критерий завершения:

```text
один набор запросов шаблонов + одна точка входа helper-ов + validate архитектуры
```

## Этап 2. Браузерная проверка

1. Выбрать минимальный инструмент headless browser.
2. Проверить загрузку страницы без console error.
3. Проверить переключение трёх режимов.
4. Проверить маршрут новичка до отчёта.
5. Проверить кнопку печати без реального открытия системного окна.
6. Добавить smoke в GitHub Actions.

Критерий завершения:

```text
критичный пользовательский путь автоматически проходит в браузере
```

## Этап 3. Отчёт как источник данных

1. Сохранять `templateId`, goal и office-сценарий.
2. Сохранять quality score и ключевые предупреждения.
3. Добавить удаление ошибочной записи.
4. Улучшить группировку аналитики по шаблону и району.
5. Добавить короткую менеджерскую сводку.

## Этап 4. Шаблоны под реальные задачи

1. Провести инвентаризацию всех шаблонов.
2. Пометить дубли и устаревшие варианты.
3. Довести локальные сценарии Борисоглебска до 20+.
4. Добавить наследство, срочную продажу, обмен, оценку, дом, участок, коммерцию.
5. Добавить брендовые сценарии СПН.
6. Добавить office-метаданные в основные пакеты.

## Этап 5. Фото-компоновки

1. `photo_left`.
2. `photo_card`.
3. `newbuild_visual`.
4. `agent_brand_photo`.
5. Проверки качества и ручной чек-лист каждого типа.

## Этап 6. Упрощение интерфейса

1. Визуально разделить цель и формат А4.
2. Сократить количество помощников, видимых одновременно.
3. Добавить итог состояния каждого шага.
4. Увеличить мелкий интерфейсный текст.
5. Улучшить мобильную работу и клавиатурную доступность.

## Этап 7. Backup и выпуск 3.86.0

1. Экспорт/импорт полного локального backup.
2. Финальная регрессия.
3. Актуализация README/help/changelog.
4. Выпуск 3.86.0.

## 7. Порядок автономной работы

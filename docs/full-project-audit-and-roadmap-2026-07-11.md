# Многосторонний аудит и автономный план развития генератора расклеек

Дата аудита: 2026-07-11

## 1. Итоговый вывод

Проект хорошо решает основную задачу специалиста по недвижимости:

```text
выбрать задачу → взять шаблон → подставить данные → проверить → напечатать → выдать задание → сохранить результат
```

Генератор уже полезен как рабочий инструмент отдела продаж, а не только как редактор листовок. Сильные стороны — печать А4, локальные сценарии Борисоглебска, контроль качества, режим новичка, задания и история результатов.

Главные ограничения текущей версии:

1. библиотека шаблонов содержит повторяющиеся `templateId`, которые могут молча перезаписывать друг друга;
2. browser smoke не фиксирует самые ранние ошибки загрузки модулей;
3. часть заявленных UX-улучшений существует только в локальной ветке и отсутствует на GitHub Pages;
4. нет полного backup/restore рабочего пространства;
5. инструмент лучше поддерживает подготовку макета, чем само полевое выполнение расклейки;
6. документация и roadmap периодически отстают от фактической ветки;
7. большое число независимых helper-модулей повышает риск скрытых конфликтов состояния.

Главный принцип дальнейшей работы:

```text
Не увеличивать количество функций без необходимости.
Устранять потери данных, сокращать время СПН и делать результат проверяемым менеджером.
```

## 2. Насколько проект решает задачи пользователя

Проект позволяет быстро выбрать задачу, локальный сценарий, формат А4, фото, QR и контакт. Для большинства типовых задач отдела продаж отдельный графический редактор не нужен.

### Работа новичка

Оценка: 8/10.

Есть безопасный фильтр, Wizard Flow, подсказки, менеджерские ограничения и защита печати. Недостаток — большое количество вспомогательных блоков всё ещё существует в общей DOM-структуре, а итог состояния шага не всегда заметен с первого взгляда.

### Работа опытного СПН

Оценка: 8/10.

Есть быстрые режимы, сохранённые макеты, профиль, фото-компоновки и ручная настройка. Не хватает полного переноса рабочего пространства между компьютерами и быстрого восстановления после очистки браузера.

### Работа менеджера

Оценка: 8/10.

### Работа расклейщика или исполнителя

### Аналитика эффективности

История сохраняет шаблон, место, формат, качество, отклики и целевые обращения. Есть фильтры, устойчивые связки и менеджерская сводка. Ограничение — локальное хранение и ручной ввод результатов.

### Печать и производство

Поддерживаются основные форматы А4, линии реза, безопасные поля, крупный телефон, QR и специализированные фото-компоновки. Нужны screenshot-регрессии и фактическая проверка нескольких принтеров.

## 4. Подтверждённые ошибки и риски

### P0. Повторяющиеся templateId

Автоматическая инвентаризация показывает четыре повторяющихся ID. Текущий загрузчик использует `Map` и выполняет `byId.set(item.id, ...)`, поэтому поздний пакет молча заменяет ранний шаблон.

Последствия:

- карточка одного пакета исчезает;
- office/portfolio-метаданные могут примениться не к тому варианту;
- аналитика смешивает разные шаблоны;
- сохранённый `templateId` перестаёт однозначно идентифицировать макет.

Решение:

- ввести явные уникальные ID или пакетный реестр алиасов;
- запретить неразрешённые дубли в validate;
- показывать источник пакета в инвентаризации;
- повторно сформировать отчёт портфеля.

### P0. Ранние runtime-ошибки не попадают в smoke

`browser-smoke.html` подписывается на `error` и `unhandledrejection` после события `iframe.load`. Ошибка модуля или раннего `DOMContentLoaded` может произойти до подписки.

Решение:

- установить ранний capture-hook в `index.html` при `?smoke=1`;
- после загрузки забрать накопленные ошибки;
- оставить поздний listener для ошибок после запуска интерфейса.

### P0. Расхождение issue и main

Issue #13 помечает keyboard/focus-функции выполненными, но комментарий сообщает, что коммиты остались только локально и не опубликованы. Значит, GitHub Pages и ветка `main` не содержат заявленного результата.

Решение:

- не отмечать функцию выполненной до появления коммита в `main`;
- перенести или повторно реализовать изменения через подключённый GitHub workflow;
- проверять наличие обязательного файла/импорта в runtime-валидаторе.

### P1. Межрежимное наследование состояния

Специализированный режим `agent_brand_photo` включает `showBrand`, но не обязан сбрасывать `colorMode=private`. При переходе из частного режима бренд может остаться скрытым.

Решение:

- явно нормализовать фирменный цвет для брендового режима;
- добавить регрессию `private → agent_brand_photo`;
- не менять частный режим без явного выбора пользователя.

### P1. Неполная сохранность данных

Профиль, макеты, избранное, UI-настройки и история распределены по нескольким ключам localStorage. Очистка браузера удаляет всё. Существующий экспорт сохраняет отдельный макет, а не рабочее пространство.

Решение:

- единый backup JSON;
- версия схемы;
- предварительная проверка импорта;
- режим `объединить` и режим `заменить`;
- подтверждение перед заменой;
- отчёт о восстановленных разделах.

### P1. Ограниченный полевой сценарий

Задание ориентировано на передачу текста, а не на выполнение с телефона.

Решение:

- компактная мобильная карточка задания;
- точки расклейки;
- план/факт листов;
- отметка выполнения;
- заметка о проблемах;
- фотоотчёт без обязательного серверного хранения;
- экспорт результата для менеджера.

### P2. Большое количество helper-модулей

`spnUiMode.js` импортирует десятки модулей, многие самостоятельно используют `DOMContentLoaded`, MutationObserver и внедрение CSS.

Решение:

- не проводить большой рефакторинг;
- ввести реестр lifecycle-фаз;
- постепенно объединять однотипные quality-actions;
- ограничивать observer конкретным контейнером;
- фиксировать владельца каждого DOM-блока в helper-map.

## 6. Правила дальнейших изменений

1. Сначала исправлять потерю данных и неоднозначность, затем улучшать внешний вид.
2. Не считать локальный коммит частью продукта, пока его нет в `main`.
3. Не удалять пользовательские данные автоматически.
4. Не менять формат localStorage без обратной совместимости.
5. Не скрывать риск шаблона без причины и замены.
6. Не добавлять глобальный click-перехват, если достаточно scoped listener.
7. Каждый новый runtime-helper должен иметь validator, smoke или ручной regression checklist.
8. Не считать browser smoke успешным без фактического запуска.
9. Для юридических, ипотечных и обещающих результат формулировок использовать manager-level.
10. Каждое улучшение связывать с конкретной ролью и измеримым сокращением ошибки или времени.

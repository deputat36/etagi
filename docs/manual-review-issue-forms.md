# GitHub-формы ручной приёмки 3.86.0

Формы помогают одинаково оформлять замечания, найденные при выполнении issues #40 и #51. Они не заменяют основные бланки и не отмечают ручные проверки автоматически.

## Печатная или viewport-регрессия

Файл формы:

```text
.github/ISSUE_TEMPLATE/print-regression.yml
```

Использовать, когда при сценариях issue #40 обнаружена воспроизводимая проблема:

- горизонтальное переполнение на ПК или телефоне 360–430 px;
- неудобный переход `К печати → Проверить`;
- обрезка телефона, QR, цены или текста;
- наложение карточек, неверный формат A4 или масштаб;
- расхождение PDF и предпросмотра;
- неприемлемый расход чернил;
- проблема конкретного принтера или драйвера.

Форма требует указать критичность, сценарий, проверенную версию, устройство, viewport, принтер, воспроизводимость, шаги, ожидаемый и фактический результат. Для экранной проблемы в поле принтера указывается `не применимо`.

После исправления issue закрывается только после повторения того же сценария на указанном устройстве или принтере. Фактический итог всё равно переносится в:

```text
docs/manual-print-acceptance-3.86.0.md
```

## Изменение чувствительного шаблона

Файл формы:

```text
.github/ISSUE_TEMPLATE/sensitive-template-change.yml
```

Использовать для отдельного решения или критичного замечания из issue #51, когда требуется изменить:

- рекламный текст;
- риск или office-уровень;
- условия применения;
- статус `test / working / deprecated`;
- конечный `replacementId`;
- временный запрет печати.

Форма содержит фиксированный список 12 чувствительных шаблонов и четыре допустимых решения менеджера. Обязательны основание решения, безопасная редакция, границы применения и план повторной проверки.

Заполненная форма не меняет данные генератора. Изменение выполняется отдельным коммитом, после чего запускаются:

```bash
npm run templates:inventory
node tools/validate-manager-sensitive-review.mjs
npm run validate
```

Итоговое решение также фиксируется в:

```text
docs/manager-sensitive-template-review-3.86.0.md
```

## Персональные данные

Обе формы напоминают не публиковать персональные данные клиентов и непубличные документы.

В issues нельзя прикладывать:

- документы клиентов;
- реальные паспортные данные;
- непубличные телефоны и адреса;
- фотографии людей без разрешения;
- внутренние банковские или юридические документы с персональными данными.

Для воспроизведения используются тестовые данные из `docs/manual-print-test-pack-3.86.0.md` и обезличенные изображения.

## Автоматическая защита

`npm run validate:ci-config` проверяет:

- наличие обеих форм и этой инструкции;
- обязательные ссылки на issues #40 и #51;
- уникальность полей формы;
- типы обязательных полей;
- наличие `required: true` у ключевых ответов;
- полный список 12 чувствительных template ID;
- запуск GitHub Actions при изменениях `.github/ISSUE_TEMPLATE/**`.

# Screenshot-регрессия печатных листов А4

Автоматический workflow создаёт PNG-снимки основных печатных сценариев после успешных `validate` и `browser-smoke`. Формат 8 на А4 дополнительно проверяется отдельным изолированным workflow, чтобы не менять уже закреплённый семисценарный комплект до подтверждения его пригодности.

## Локальные команды

Основная матрица:

```bash
npm run screenshots:print
```

Для одного сценария основной матрицы используется переменная:

```bash
PRINT_SCREENSHOT_SCENARIO=six-economy npm run screenshots:print
```

Отдельная проверка 8 на А4:

```bash
node tools/validate-eight-print-screenshot.mjs
node tools/run-eight-print-screenshot.mjs
```

## Надёжный захват через Chrome DevTools Protocol

Оба runner запускают Chrome с `--remote-debugging-pipe` и управляют страницей через Chrome DevTools Protocol.

Последовательность захвата:

1. открыть страницу сценария;
2. опрашивать `captureStatus` через `Runtime.evaluate`;
3. дождаться фактического `data-status="passed"`;
4. только после этого вызвать `Page.captureScreenshot`;
5. сохранить PNG и метаданные времени ожидания.

Фиксированный virtual-time budget и CLI-снимок «по таймеру» не используются. Это защищает от PNG, снятого до завершения рендера компоновки или проверки контактов.

Ошибки чтения и записи CDP pipe обрабатываются общим guard-модулем. В частности, `ECONNRESET` и `EPIPE` превращаются в штатный reject активной CDP-команды с сохранением направления `read / write` и кода ошибки. Runner может выполнить предусмотренную вторую попытку, а если обе попытки неуспешны, записывает failure-log для artifact. Проверки `captureStatus`, размера PNG и содержимого листа при этом не ослабляются.

## Контролируемая проверка двойного сбоя

Для проверки полного аварийного пути основной матрицы используется отдельная команда:

```bash
npm run test:cdp-failure-artifact
```

Тест запускает штатный `tools/run-print-screenshots.mjs` для `one-showcase` через изолированный fake Chrome. Поддельный процесс успешно отвечает только на `--version`, а при реальном запуске дважды завершается до ответа CDP. Проверка считается успешной только когда:

- screenshot runner выполнил обе предусмотренные попытки;
- runner завершился ненулевым кодом;
- создан `print-screenshots-failure.log` с записью второй попытки и диагностикой fault injection;
- ошибочный `one-showcase.png` и его JSON-метаданные не созданы.

GitHub Actions выполняет это отдельным job `cdp-failure-artifact` после `validate` и загружает одноимённый artifact. Обычные screenshot-проверки не используют fake Chrome и не зависят от тестовой ошибки.

## Основная изолированная матрица GitHub Actions

В основном workflow каждый сценарий запускается в отдельном чистом job Chrome. Это исключает влияние предыдущего браузерного процесса и сразу показывает, какой конкретный лист не сформировался.

Matrix содержит:

```text
one-no-photo
two-big-phone
three-tearoffs
one-showcase
two-photo
four-contacts
six-economy
```

Каждый job создаёт PNG и одноимённый JSON с размером, номером попытки, методом захвата и временем ожидания. Частичные artifacts хранятся 1 день.

После успешного завершения всех семи job команда:

```bash
npm run screenshots:manifest
```

проверяет комплектность, собирает `manifest.json` и формирует единый artifact `print-screenshots`.

## Отдельная проверка 8 на А4

Workflow `Validate 8 on A4` запускает один независимый job `eight-print-screenshot`. Он сначала выполняет статический контракт, затем создаёт `eight-economy.png` и `eight-economy.json`.

Успешный artifact называется:

```text
print-screenshot-eight-economy
```

При ошибке загружается `print-screenshot-failure-eight-economy` с файлом `eight-print-screenshot-failure.log`.

## Что проверяется до снимка

Harness не создаёт PNG вслепую. Для каждого сценария он проверяет:

- нужное количество макетов на листе;
- наличие или отсутствие фото;
- отсутствие класса `overflow`;
- нахождение карточек и контактного блока внутри границ;
- готовность страницы через `data-status="passed"`.

Для `two-big-phone.png` дополнительно проверяется крупный телефон и прямое редактирование заголовка на макете.

Для `three-tearoffs.png` дополнительно проверяется:

- ровно три макета на A4;
- минимально читаемый размер телефона;
- светлая экономная контактная зона;
- наличие восьми отрывных полос в каждой карточке;
- телефон внутри каждой отрывной полосы;
- границы общего блока и каждой отдельной полосы;
- отсутствие выхода контактов и отрывных элементов за карточку.

Для `four-contacts.png` дополнительно проверяется:

- автоматическое включение `colorMode = economy` при выборе режима `Подъезд`;
- светлый фон контактной зоны вместо повторяющейся сплошной тёмной заливки;
- видимая цветная рамка;
- тёмный контрастный телефон;
- сохранение четырёх макетов на А4.

Для `six-economy.png` дополнительно проверяется:

- ровно шесть макетов в сетке 2×3;
- отсутствие фотографий и отрывных полос;
- экономный цветовой режим;
- минимальный размер телефона 18 px;
- минимальный размер заголовка 14 px;
- наличие полного телефона и имени специалиста в каждой карточке;
- отсутствие overflow и выхода контактной зоны за границы.

Для `eight-economy.png` дополнительно проверяется:

- ровно восемь макетов в сетке 2×4;
- отсутствие фотографий и отрывных полос;
- экономный цветовой режим;
- минимальный размер телефона 16 px;
- минимальный размер заголовка 12 px;
- наличие полного телефона и имени специалиста в каждой карточке;
- нахождение каждой карточки внутри A4;
- отсутствие overflow и выхода контактной зоны за границы;
- светлая контактная зона с видимой рамкой.

## Итоговые artifacts

Основной workflow создаёт `print-screenshots`. В комплекте находятся семь PNG, семь служебных JSON и общий `manifest.json`. Срок хранения — 7 дней.

Отдельный workflow создаёт `print-screenshot-eight-economy` с PNG и JSON формата 8 на А4. Срок хранения — 7 дней.

## Ручная визуальная проверка

После изменений печатных CSS, рендера, компоновок, размеров телефона, контактов или цветности:

1. открыть artifacts последнего workflow;
2. проверить, что лист заполнен без обрезки по краям;
3. сравнить размеры заголовка и телефона между сценариями;
4. убедиться, что контакты не перекрывают текст и не выходят из карточек;
5. проверить `three-tearoffs.png`: три полосы должны быть одинаковыми, телефон и отрывные элементы — читаемыми и не обрезанными;
6. проверить фото: `Витрина` должна оставаться крупной, `С фото` — читабельной при двух карточках;
7. проверить `four-contacts.png`: фон контактов должен быть светлым, рамка заметной, телефон читаемым;
8. проверить `six-economy.png`: шесть карточек должны быть одинаковыми, а заголовок, телефон и имя СПН — различимыми без увеличения;
9. проверить `eight-economy.png`: восемь карточек должны быть одинаковыми, не выходить за лист, а телефон и заголовок — оставаться различимыми;
10. убедиться, что экономия не превратила макет в бледный или чёрно-белый;
11. при подозрении на регрессию выполнить ручную печать в PDF по чек-листу полного сценария.

## Приватность

Используются только синтетические данные. Реальные фотографии, телефоны клиентов и пользовательский localStorage в artifacts не попадают.

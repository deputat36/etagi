# Screenshot-регрессия печатных листов А4

Автоматический workflow создаёт PNG-снимки основных печатных сценариев после успешных `validate` и `browser-smoke`.

## Локальная команда

```bash
npm run screenshots:print
```

Локально команда может сформировать весь набор. Для одного сценария используется переменная:

```bash
PRINT_SCREENSHOT_SCENARIO=three-tearoffs npm run screenshots:print
```

## Надёжный захват через Chrome DevTools Protocol

Runner запускает Chrome с `--remote-debugging-pipe` и управляет страницей через Chrome DevTools Protocol.

Последовательность захвата:

1. открыть страницу сценария;
2. опрашивать `captureStatus` через `Runtime.evaluate`;
3. дождаться фактического `data-status="passed"`;
4. только после этого вызвать `Page.captureScreenshot`;
5. сохранить PNG и метаданные времени ожидания.

Фиксированный virtual-time budget и CLI-снимок «по таймеру» не используются. Это защищает от PNG, снятого до завершения `FileReader`, рендера компоновки или проверки контактов.

Ошибки чтения и записи CDP pipe обрабатываются общим guard-модулем. В частности, `ECONNRESET` и `EPIPE` превращаются в штатный reject активной CDP-команды с сохранением направления `read / write` и кода ошибки. Print runner может выполнить предусмотренную вторую попытку, а если обе попытки неуспешны, общий `catch` записывает `print-screenshots-failure.log` для failure-artifact. Проверки `captureStatus`, размера PNG и содержимого листа при этом не ослабляются. Тот же guard защищает browser smoke от аварийного необработанного события stream.

## Контролируемая проверка двойного сбоя

Для проверки полного аварийного пути используется отдельная команда:

```bash
npm run test:cdp-failure-artifact
```

Тест запускает штатный `tools/run-print-screenshots.mjs` для `one-showcase` через изолированный fake Chrome. Поддельный процесс успешно отвечает только на `--version`, а при реальном запуске дважды завершается до ответа CDP. Проверка считается успешной только когда:

- screenshot runner выполнил обе предусмотренные попытки;
- runner завершился ненулевым кодом;
- создан `print-screenshots-failure.log` с записью второй попытки и диагностикой fault injection;
- ошибочный `one-showcase.png` и его JSON-метаданные не созданы.

GitHub Actions выполняет это отдельным job `cdp-failure-artifact` после `validate` и загружает одноимённый artifact. Job сам остаётся зелёным, потому что ожидаемое падение вложенного screenshot-runner проверяется тестовым скриптом. Обычная screenshot matrix не использует fake Chrome и не зависит от тестовой ошибки.

## Изолированная матрица GitHub Actions

В CI каждый сценарий запускается в отдельном чистом job Chrome. Это исключает влияние предыдущего браузерного процесса и сразу показывает, какой конкретный лист не сформировался.

Matrix содержит:

```text
one-no-photo
two-big-phone
three-tearoffs
one-showcase
two-photo
four-contacts
```

Каждый job создаёт PNG и одноимённый JSON с размером, номером попытки, методом захвата и временем ожидания. Частичные artifacts хранятся 1 день.

После успешного завершения всех шести job команда:

```bash
npm run screenshots:manifest
```

проверяет комплектность, собирает `manifest.json` и формирует единый artifact `print-screenshots`.

## Сценарии

| Файл | Проверяемый сценарий |
|---|---|
| `one-no-photo.png` | 1 макет на А4 без фото |
| `two-big-phone.png` | 2 макета на А4 с крупным телефоном |
| `three-tearoffs.png` | 3 горизонтальных макета на А4 с контактами и отрывными полосами |
| `one-showcase.png` | 1 макет на А4 в режиме `Витрина` |
| `two-photo.png` | 2 макета на А4 в режиме `С фото` |
| `four-contacts.png` | 4 макета на А4 без наложений и с облегчённой цветной контактной зоной |

Файлы собираются в:

```text
artifacts/print-screenshots/
```

## Что проверяется до снимка

Harness не создаёт PNG вслепую. Для каждого сценария он проверяет:

- нужное количество макетов на листе;
- наличие или отсутствие фото;
- фактическую отрисовку загруженного изображения;
- отсутствие класса `overflow`;
- нахождение контактного блока внутри карточки;
- готовность страницы через `data-status="passed"`.

Для `two-big-phone.png` дополнительно проверяется крупный телефон и прямое редактирование заголовка на макете.

Для `three-tearoffs.png` дополнительно проверяется:

- ровно три макета на A4;
- минимально читаемый размер телефона;
- светлая экономная контактная зона;
- наличие восьми отрывных полос в каждой карточке;
- телефон внутри каждой отрывной полосы;
- границы общего блока и каждой отдельной полосы;
- отсутствие выхода контактов и отрывных элементов за карточку.

Для `four-contacts.png` дополнительно проверяется:

- автоматическое включение `colorMode = economy` при выборе режима `Подъезд` из фирменного цвета;
- светлый фон контактной зоны вместо повторяющейся сплошной тёмной заливки;
- видимая цветная рамка;
- тёмный контрастный телефон;
- сохранение четырёх макетов на А4.

При ошибке конкретный matrix job загружает artifact `print-screenshot-failure-<scenario>` с файлом `print-screenshots-failure.log`.

## Итоговый artifact

После успешного запуска создаётся:

```text
print-screenshots
```

Он содержит шесть PNG, шесть служебных JSON и общий `manifest.json`. Срок хранения — 7 дней.

## Ручная визуальная проверка

После изменений печатных CSS, рендера, компоновок, размеров телефона, контактов или цветности:

1. открыть artifact `print-screenshots` последнего workflow;
2. проверить, что лист заполнен без обрезки по краям;
3. сравнить размеры заголовка и телефона между сценариями;
4. убедиться, что контакты не перекрывают текст и не выходят из карточек;
5. проверить `three-tearoffs.png`: три полосы должны быть одинаковыми, телефон и отрывные элементы — читаемыми и не обрезанными;
6. проверить фото: `Витрина` должна оставаться крупной, `С фото` — читабельной при двух карточках;
7. проверить `four-contacts.png`: фон контактов должен быть светлым, рамка заметной, телефон читаемым;
8. убедиться, что экономия не превратила макет в бледный или чёрно-белый;
9. при подозрении на регрессию выполнить ручную печать в PDF по чек-листу полного сценария.

## Приватность

Используются только синтетические данные и SVG-заглушка здания. Реальные фотографии, телефоны клиентов и пользовательский localStorage в artifact не попадают.

# Screenshot-регрессия печатных листов А4

Автоматический workflow создаёт PNG-снимки основных печатных сценариев после успешных `validate` и `browser-smoke`.

## Локальная команда

```bash
npm run screenshots:print
```

Локально команда может сформировать весь набор. Для одного сценария используется переменная:

```bash
PRINT_SCREENSHOT_SCENARIO=four-contacts npm run screenshots:print
```

## Изолированная матрица GitHub Actions

В CI каждый сценарий запускается в отдельном чистом job Chrome. Это исключает влияние предыдущего браузерного процесса и сразу показывает, какой конкретный лист не сформировался.

Matrix содержит:

```text
one-no-photo
two-big-phone
one-showcase
two-photo
four-contacts
```

Каждый job создаёт PNG и одноимённый JSON с размером, номером попытки и virtual-time budget. Частичные artifacts хранятся 1 день.

После успешного завершения всех пяти job команда:

```bash
npm run screenshots:manifest
```

проверяет комплектность, собирает `manifest.json` и формирует единый artifact `print-screenshots`.

## Сценарии

| Файл | Проверяемый сценарий |
|---|---|
| `one-no-photo.png` | 1 макет на А4 без фото |
| `two-big-phone.png` | 2 макета на А4 с крупным телефоном |
| `one-showcase.png` | 1 макет на А4 в режиме `Витрина` |
| `two-photo.png` | 2 макета на А4 в режиме `С фото` |
| `four-contacts.png` | 4 макета на А4 без наложений и с облегчённой цветной контактной зоной |

Файлы собираются в:

```text
artifacts/print-screenshots/
```

## Что проверяется до снимка

Harness не создаёт PNG вслепую. Для каждого сценария он проверяет:

- нужное количество макетов на листе;
- наличие или отсутствие фото;
- фактическую отрисовку загруженного изображения;
- отсутствие класса `overflow`;
- нахождение контактного блока внутри карточки;
- крупный телефон в сценарии `2 на А4`;
- готовность страницы через `data-status="passed"`.

Для `four-contacts.png` дополнительно проверяется:

- автоматическое включение `colorMode = economy` при выборе режима `Подъезд` из фирменного цвета;
- светлый фон контактной зоны вместо повторяющейся сплошной тёмной заливки;
- видимая цветная рамка;
- тёмный контрастный телефон;
- сохранение четырёх макетов на А4.

При ошибке конкретный matrix job загружает artifact `print-screenshot-failure-<scenario>` с файлом `print-screenshots-failure.log`.

## Итоговый artifact

После успешного запуска создаётся:

```text
print-screenshots
```

Он содержит пять PNG, пять служебных JSON и общий `manifest.json`. Срок хранения — 7 дней.

## Ручная визуальная проверка

После изменений печатных CSS, рендера, компоновок, размеров телефона, контактов или цветности:

1. открыть artifact `print-screenshots` последнего workflow;
2. проверить, что лист заполнен без обрезки по краям;
3. сравнить размеры заголовка и телефона между сценариями;
4. убедиться, что контакты не перекрывают текст и не выходят из карточек;
5. проверить фото: `Витрина` должна оставаться крупной, `С фото` — читабельной при двух карточках;
6. проверить `four-contacts.png`: фон контактов должен быть светлым, рамка заметной, телефон читаемым;
7. убедиться, что экономия не превратила макет в бледный или чёрно-белый;
8. при подозрении на регрессию выполнить ручную печать в PDF по чек-листу полного сценария.

## Приватность

Используются только синтетические данные и SVG-заглушка здания. Реальные фотографии, телефоны клиентов и пользовательский localStorage в artifact не попадают.

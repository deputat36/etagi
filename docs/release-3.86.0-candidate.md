# Релиз-кандидат 3.86.0

Статус: DRAFT

Целевая версия: 3.86.0

Текущая опубликованная версия: 3.85.0

Причина статуса `DRAFT`: автоматические проверки пройдены, но ручная проверка доступа к печати на рабочем ПК и телефоне, физическая печатная приёмка и менеджерская проверка чувствительных шаблонов ещё не подтверждены.

## Что войдёт в 3.86.0

### Рабочий путь СПН

- режимы `Новичок`, `Быстро` и `Расширенно`;
- пошаговый Wizard Flow;
- быстрый переход к проверке печати;
- сохранение профиля и макетов;
- полный локальный backup;
- мобильный режим выполнения расклейки;
- задание, отчёт и аналитика после печати.

### Шаблоны и безопасность

- расширенная библиотека сценариев Борисоглебска;
- office-уровни, риски и заметки менеджера;
- жизненный цикл `working / test / deprecated`;
- уникальные итоговые `templateId` и алиасы исторических дублей;
- контролируемые ипотечные, юридические, частные и A/B-сценарии;
- менеджерский notice для чувствительных формулировок;
- обязательный бланк `docs/manager-sensitive-template-review-3.86.0.md`;
- evidence-пакет `docs/manager-sensitive-template-evidence-3.86.0.md` для 12 шаблонов;
- автоматическая сверка `node tools/validate-manager-sensitive-review.mjs`.

### Сохранность данных

- схема и диагностика JSON-макета;
- защита от повторного и ошибочного импорта;
- явный выбор при совпадении названия;
- резерв перед очисткой, импортом и загрузкой;
- отмена последнего разрушительного действия;
- подтверждение удаления именованного макета;
- восстановление последнего удалённого макета без перезаписи соседней записи.

### Фото и печать

- официальные форматы `1, 2, 3, 4, 6 и 8 макетов на А4`;
- блокировка неподдерживаемых 10/12;
- компактные профили форматов 3 и 8;
- автоматическое отключение отрывных полос для 6/8;
- единый browser-контракт телефона, overflow, контактов и полос;
- screenshot-сценарии 1/2/3/4/6 и отдельный 8-на-А4;
- режимы `photo_left`, `photo_card`, `newbuild_visual` и `agent_brand_photo`;
- безопасное отображение планировок и портретов;
- экономный цвет без чрезмерных сплошных заливок.

### Контроль и аналитика

- проверки качества и безопасные действия исправления;
- защита печати новичка;
- история отчётов и CSV;
- устойчивые связки `templateId + место`;
- сводка менеджеру за период;
- подтверждение удаления ошибочного отчёта.

### Надёжность разработки

- единый Promise-кеш шаблонов;
- единый asset-version из `package.json`;
- команда `npm run assets:stamp`;
- общий `npm run validate`;
- настоящий browser smoke;
- UI actions smoke и диагностические artifacts;
- семь независимых печатных screenshot-сценариев через Chrome DevTools Protocol;
- отдельный screenshot формата 8;
- единый browser coverage форматов 1/2/3/4/6/8;
- `npm run release:status` и подробный режим;
- `npm run test:release-readiness-states`.

## Версии при переходе в READY

`package.version` и поле `Текущая опубликованная версия` имеют разный смысл:

- `package.version` меняется на `3.86.0`, когда полностью проверенный пакет готов к публикации;
- `Текущая опубликованная версия` остаётся `3.85.0`, пока 3.86.0 фактически не опубликована.

Нельзя заранее называть 3.86.0 опубликованной ради прохождения validator.

## Порядок перевода в READY

1. Выполнить сценарий 0 на рабочем ПК и реальном телефоне.
2. Заполнить ручную печатную приёмку всех форматов.
3. Заполнить менеджерскую проверку и закрыть issue #51.
4. Выполнить `node tools/validate-manager-sensitive-review.mjs`.
5. Проверить `npm run release:status -- --strict`: до закрытия блокеров ожидается код `2`.
6. Изменить `Статус: DRAFT` на `Статус: READY`.
7. Обновить `package.json` до `3.86.0`, не меняя опубликованную версию.
8. Выполнить `npm run assets:stamp`.
9. Добавить раздел `## 3.86.0` в `docs/changelog.md`.
10. Выполнить `npm run test:release-readiness-states`.
11. Получить код `0` от `npm run release:status -- --strict`.
12. Запустить полный `npm run validate`.
13. Подтвердить browser smoke и всю screenshot matrix.
14. Только после этого создавать релиз.
15. После фактической публикации обновить поле опубликованной версии отдельным PR.

До выполнения этих шагов повышение версии и финальный changelog считаются преждевременными.

# Runtime registry модулей

Реестр нужен для безопасного перехода от большого связанного runtime к контролируемой архитектуре. Он не меняет порядок загрузки и не отключает функции, а фиксирует фактический граф модулей и используемые ими ресурсы.

## Файлы

```text
data/runtime-modules.json
tools/build-runtime-module-registry.mjs
tools/validate-runtime-module-registry.mjs
```

## Что фиксируется

`data/runtime-modules.json` содержит:

- все локальные `<script type="module">` из `index.html`;
- все достижимые статические, side-effect и dynamic imports;
- обратные связи `importedBy`;
- назначение каждой точки входа;
- домен модуля;
- владельцев и потребителей DOM ID;
- статические и динамические ключи `localStorage` / `sessionStorage`;
- экземпляры `MutationObserver`, `ResizeObserver`, `IntersectionObserver` и `PerformanceObserver`;
- ID стилей, которые модули внедряют через JavaScript;
- итоговые счётчики по модулям, связям и ресурсам.

## DOM ownership

Для каждого DOM ID сохраняются:

- `declaredInIndex` — существует ли узел в исходном `index.html`;
- `createdBy` — какие модули создают ID во время выполнения;
- `boundBy` — какие модули назначают обработчики;
- `mutatedBy` — какие модули изменяют состояние или содержимое;
- `referencedBy` — какие модули обращаются к ID;
- `owners` — основной список владельцев по правилу `create → bind/mutate → reference`;
- `ownershipBasis` — основание выбора владельца.

Это машинная инвентаризация, а не разрешение двум модулям одновременно изменять один узел. Совпадающие владельцы используются как сигнал для следующего архитектурного аудита.

## Storage ownership

Для каждого ключа фиксируются:

- область хранения;
- точный ключ либо нормализованное динамическое выражение;
- операции `getItem`, `setItem`, `removeItem`;
- модули, которые используют ключ.

Динамические выражения отмечаются `dynamic: true` и не смешиваются со статическими ключами.

## Observers и injected styles

Для observers фиксируются модуль, тип, количество экземпляров, вызовы `observe` и `disconnect`.

Для injected styles фиксируются ID, модули-владельцы и модули, которые непосредственно создают style-узел. Это позволяет отдельно искать постоянные observers и стили, которые можно безопасно перенести в CSS.

## Обновление

После добавления, удаления или переноса runtime-модуля, DOM ID, storage key, observer или injected style выполнить:

```bash
npm run runtime:registry
npm run validate:runtime-modules
```

Общий `npm run validate` также запускает проверку реестра.

## Что запрещает validator

- отсутствующий или некорректный JSON;
- пропущенную точку входа `index.html`;
- новый импорт или ресурс без обновления реестра;
- запись о несуществующем файле;
- повторяющиеся module paths, DOM ID, storage records и style IDs;
- несогласованные связи `imports` и `importedBy`;
- ссылку владельца на отсутствующий runtime-модуль;
- неверное правило определения DOM owner;
- неизвестный observer или storage operation;
- несогласованные итоговые счётчики;
- неполный side-effect bundle `spnUiMode.js`;
- query/hash в каноническом пути модуля.

## Границы

Парсер разрешает простые строковые константы и статические селекторы. Сложные вычисляемые ключи сохраняются как динамические выражения. Реестр показывает фактические обращения, но не заменяет browser smoke и ручную проверку пользовательского сценария.

Реестр не является основанием для массового объединения или удаления модулей. Любой рефакторинг выполняется отдельным небольшим PR с browser smoke и профильными проверками.

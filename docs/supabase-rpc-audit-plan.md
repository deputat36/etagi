# План аудита Supabase RPC

Дата подготовки: 2026-06-29.

Проект Supabase: `deputat36's Project`.
Project ref: `ofewxuqfjhamgerwzull`.

## Зачем нужен этот документ

В Supabase-проекте есть несколько рабочих направлений: CRM РА Лидер, навигатор сделок, Паркет36 и legacy-таблицы. Security advisors показывают много предупреждений по `SECURITY DEFINER` RPC-функциям в `public`, доступным роли `authenticated`.

Это не означает, что все функции нужно срочно отключать. Для навигатора сделок RPC-слой часто нужен, чтобы централизовать проверки ролей и доступа к сделке. Но каждая такая функция должна иметь понятный статус: кто её вызывает, зачем она нужна, какие права внутри проверяются и кому разрешён `EXECUTE`.

## Главный принцип

Не делать массовых `REVOKE` и не переводить функции на `SECURITY INVOKER` вслепую.

Сначала нужно составить инвентарь, затем закрывать только те функции, у которых понятен риск и понятна замена.

## Категории RPC

### 1. Фронтендовые read RPC

Функции, которые реально нужны интерфейсу для отображения списков, карточек, очередей, сводок и дашбордов.

Примеры:

- `nav_v2_get_dashboard`;
- `nav_v2_get_deals_list`;
- `nav_v2_get_deal_card`;
- `nav_v2_get_deal_card_lite`;
- `nav_v2_get_lawyer_queue`;
- `nav_v2_get_lawyer_review_summary`;
- `nav_v2_get_deal_responsibility_snapshot`;
- `nav_v2_get_handoff_scores`.

Проверить:

- вызывается ли функция фронтендом;
- нужна ли она всем авторизованным или только определённым ролям;
- фильтрует ли она данные по текущему пользователю;
- не отдаёт ли лишние телефоны, ФИО, комментарии, документы и служебные поля;
- есть ли лимиты на списки.

### 2. Фронтендовые write RPC

Функции, которые меняют сделки, задачи, документы, риски, комментарии, статусы и участников.

Примеры:

- `nav_v2_save_wizard_result`;
- `nav_v2_add_comment`;
- `nav_v2_add_task`;
- `nav_v2_add_document`;
- `nav_v2_add_expense`;
- `nav_v2_add_risk`;
- `nav_v2_update_deal_status`;
- `nav_v2_update_task_status`;
- `nav_v2_update_task_due_date`;
- `nav_v2_update_document_status`;
- `nav_v2_update_document_assignment`;
- `nav_v2_update_document_workflow`;
- `nav_v2_update_deal_parties`;
- `nav_v2_return_spn_rework`;
- `nav_v2_submit_spn_rework`.

Проверить:

- есть ли проверка доступа к конкретной сделке;
- есть ли проверка роли пользователя;
- нельзя ли передать чужой `p_deal_id`, `p_task_id`, `p_document_id` и изменить чужую сделку;
- валидируются ли статусы и переходы;
- логируются ли важные изменения;
- не используются ли параметры пользователя вместо `auth.uid()` там, где это опасно.

### 3. Access helper RPC

Функции, которые возвращают роль, активность пользователя или право просмотра/редактирования.

Примеры:

- `nav_v2_can_view_deal`;
- `nav_v2_can_edit_deal`;
- `nav_v2_is_active_user`;
- `nav_v2_is_owner_or_admin`;
- `nav_v2_my_role`;
- `nav_current_role`;
- `nav_is_admin`.

Проверить:

- должны ли они быть доступны фронтенду напрямую;
- можно ли заменить часть вызовов на внутреннее использование внутри RPC/политик;
- не раскрывают ли они лишнюю информацию о ролях;
- используют ли они `auth.uid()` вместо доверия к переданному `p_uid`, если функция вызывается внешним пользователем.

### 4. Admin RPC

Функции управления пользователями, профилями и административными данными.

Примеры:

- `nav_v2_list_users`;
- `nav_v2_link_user_by_email`;
- `nav_v2_update_user_profile`.

Проверить:

- закрыт ли вызов от обычного `authenticated`;
- проверяется ли роль admin/owner внутри функции;
- защищена ли функция от самоповышения роли;
- логируются ли изменения прав.

### 5. Demo и служебные RPC

Функции посева, очистки демо-данных и диагностики.

Примеры:

- `nav_v2_seed_demo_data`;
- `nav_v2_clear_demo_data`;
- `nav_v2_get_access_audit`;
- `nav_v2_get_security_hardening_health`;
- `nav_v2_get_rpc_grant_health`;
- `nav_v2_get_internal_rpc_lockdown_health`;
- `nav_v2_get_storage_security_health`;
- `nav_v2_get_rls_policy_health`;
- `nav_v2_get_index_health`;
- `nav_v2_get_data_integrity_health`;
- `nav_v2_get_frontend_rpc_coverage_health`;
- `nav_v2_get_team_profile_quality_health`.

Проверить:

- кому они реально нужны;
- не должны ли они быть доступны только admin или service role;
- не раскрывают ли внутреннюю структуру базы обычным пользователям;
- можно ли оставить наружу только агрегированный health status без деталей.

## Таблица инвентаризации RPC

Для каждой функции нужно заполнить строки такого вида:

| RPC | Категория | Кто вызывает | Нужен фронтенду | SECURITY DEFINER нужен | EXECUTE для authenticated нужен | Внутри есть проверка роли | Внутри есть проверка доступа к объекту | Решение |
|---|---|---|---|---|---|---|---|---|
| `nav_v2_get_deal_card` | read | frontend | да | уточнить | вероятно да | проверить | проверить | оставить/сузить |
| `nav_v2_add_task` | write | frontend | да | возможно | да, если есть проверка | проверить | обязательно | аудит тела функции |
| `nav_v2_list_users` | admin | admin UI | только admin | возможно | нет для обычных | обязательно | не применимо | ограничить |
| `nav_v2_seed_demo_data` | demo | служебно | нет | нет/уточнить | нет | обязательно | не применимо | закрыть от обычных |

## Порядок безопасной работы

1. Не менять grants до подтверждения, что функция не используется фронтендом.
2. Найти вызовы RPC во фронтенде GitHub-проекта навигатора.
3. Разделить функции на read, write, access helper, admin, demo, health.
4. Для write RPC проверить тело функции и внутренние проверки.
5. Для admin/demo RPC сначала закрыть доступ от обычного `authenticated`, если это не ломает интерфейс.
6. После каждой группы изменений запускать Supabase security advisors.
7. Каждую правку оформлять отдельной миграцией с понятным именем.
8. Не смешивать в одной миграции `leader_*`, `nav_*`, `parket_*`.

## Что считать красными флагами

- функция принимает `p_uid` и доверяет ему как текущему пользователю;
- функция меняет запись по переданному ID без проверки доступа к родительской сделке;
- функция с `SECURITY DEFINER` доступна `authenticated`, но не проверяет роль;
- demo/admin функция доступна всем авторизованным;
- health RPC раскрывает список таблиц, политик, индексов или grants обычному пользователю;
- write RPC не пишет событие или audit trail для важных действий.

## Свежая миграционная зона внимания

По состоянию на 2026-06-29 в истории миграций есть свежая миграция:

```text
20260629133802 nav_v2_frontend_coverage_include_status_and_handoff_rpcs
```

Это значит, что покрытие frontend RPC продолжает развиваться. Перед ограничением `EXECUTE` нужно особенно внимательно проверить функции статусов, handoff/responsibility snapshot и frontend coverage health, чтобы не сломать рабочий интерфейс.

## Результат аудита

Итогом должна быть не только серия миграций, но и список функций в одном из статусов:

- `оставить доступ authenticated` — функция нужна фронтенду и сама проверяет доступ;
- `оставить только admin` — функция нужна только администраторам;
- `оставить только service role` — функция служебная;
- `перенести в private schema` — функция не должна быть публичным RPC;
- `переписать` — функция нужна, но текущая проверка доступа недостаточна;
- `удалить позже` — функция legacy и не используется.

## Что не делать

- Не отключать все предупреждения advisor одной массовой миграцией.
- Не переносить legacy и v2 функции одновременно без плана отката.
- Не менять RLS и grants для CRM РА Лидер во время задачи по навигатору сделок.
- Не считать отсутствие warning гарантией безопасности.
- Не считать наличие warning автоматической ошибкой: важен контекст функции.

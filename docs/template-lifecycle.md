# Жизненный цикл шаблонов

Документ объясняет, как поддерживать библиотеку без удаления сохранённых макетов и без накопления непонятных дублей.

## Реестр

Статусы хранятся отдельно от текстов шаблонов:

```text
data/template_portfolio_status.json
```

Это позволяет:

- не редактировать десятки JSON-объектов ради служебной отметки;
- сохранить совместимость старых макетов по `templateId`;
- централизованно менять статус;
- указать рекомендуемую замену;
- использовать один статус в карточках, поиске и инвентаризации.

## Статусы

### working

Основной рабочий шаблон.

Использовать, если:

- сценарий не дублирует более точный вариант;
- формулировки актуальны;
- шаблон можно применять в текущем офисном процессе;
- при необходимости заданы office-уровень и риск.

### test

Тестовый или требующий актуальной проверки шаблон.

Примеры:

- A/B-вариант;
- ипотечные или государственные условия, которые могут меняться;
- частный макет без бренда;
- новая формулировка, по которой ещё нет отчётов;
- шаблон для контролируемого сравнения.

Тестовый шаблон не должен выдаваться новичку как основной рекомендованный вариант без проверки менеджером.

### deprecated

Устаревший шаблон или смысловой дубль.

Он не удаляется, потому что его `templateId` может храниться:

- в именованных макетах;
- в последнем состоянии браузера;
- в истории отчётов;
- в старом JSON-экспорте.

Карточка показывает:

```text
Устарел
Причина
Используйте: replacementId
```

## Формат реестра

```json
{
  "version": 1,
  "defaultStatus": "working",
  "packageDefaults": {
    "templates_ab_tests.json": {
      "status": "test",
      "reason": "Пакет предназначен для контролируемых A/B-тестов."
    }
  },
  "templates": {
    "old_template_id": {
      "status": "deprecated",
      "replacementId": "new_template_id",
      "reason": "Новый вариант точнее описывает рабочий сценарий."
    }
  }
}
```

## Правила replacementId

Для `deprecated` обязательно указать конечный рабочий `replacementId`.

Запрещено:

```text
шаблон → сам на себя
A → B → A
A deprecated → B deprecated → C working
replacementId, которого нет в библиотеке
```

Правильно:

```text
A deprecated → C working
B deprecated → C working
```

То есть каждый устаревший вариант сразу ведёт на конечную актуальную замену.

## Когда помечать устаревшим

Шаблон можно считать смысловым дублем, если совпадают:

- аудитория;
- задача;
- основной повод позвонить;
- ключевой текст;
- формат использования.

Небольшая разница заголовка или декоративного оформления не является отдельным рабочим сценарием.

Шаблон не нужно помечать устаревшим, если отличается хотя бы один важный фактор:

- другая аудитория;
- другая цель;
- другой уровень риска;
- другой формат А4;
- фото вместо текстового объявления;
- локальный район или ЖК;
- отдельная бизнес-ситуация.

## Карточки интерфейса

`assets/js/templates.js` объединяет шаблон с реестром и добавляет:

```json
"portfolio": {
  "status": "working",
  "reason": "",
  "replacementId": ""
}
```

`assets/js/spnTemplateCardBadges.js` показывает:

```text
Тест
Устарел
```

Для устаревшего шаблона причина имеет приоритет над обычной office-подсказкой.

Все тексты из реестра экранируются перед вставкой в HTML.

## Автоматическая проверка

```bash
npm run validate:template-portfolio
```

Проверка контролирует:

- формат реестра и `version`;
- допустимые статусы;
- существование пакетов и templateId;
- понятную причину статуса;
- наличие замены для deprecated;
- отсутствие циклов и устаревших цепочек;
- подключение реестра в загрузчике;
- отображение статусов в карточках.

## Общая инвентаризация

```bash
npm run templates:inventory
```

Команда создаёт:

```text
docs/template-portfolio-inventory.generated.md
```

Отчёт содержит:

- количество working, test и deprecated;
- office-покрытие каждого пакета;
- список устаревших шаблонов и замен;
- тестовые варианты;
- одинаковые заголовки;
- вероятные смысловые дубли;
- шаблоны без office-метаданных.

## Безопасный порядок изменений

1. Запустить инвентаризацию.
2. Сравнить вероятные дубли вручную.
3. Выбрать конечный рабочий вариант.
4. Добавить статус и причину в реестр.
5. Не удалять старый JSON-объект.
6. Запустить `npm run validate`.
7. Пройти browser smoke, если менялся загрузчик или карточки.
8. Проверить карточку в режимах `Новичок`, `Быстро`, `Расширенно`.

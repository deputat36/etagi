# Реестр действий интерфейса

Machine-readable источник:

```text
data/ui-actions.json
```

Проверка:

```bash
npm run test:ui-actions
npm run smoke:browser
npm run smoke:ui-actions
npm run validate
```

## Назначение

Реестр отвечает на четыре вопроса по каждой устойчивой команде интерфейса:

1. какой selector запускает действие;
2. какой JS-файл является владельцем;
3. какой пользовательский эффект ожидается;
4. каким способом действие проверяется.

Обычные поля ввода и чекбоксы не перечисляются как отдельные команды. Их связь со state уже контролируется массивами `fields` и `checks` в `assets/js/app.js` и проверкой `validate:assets`.

В реестр входят:

- все статические кнопки основного `index.html`;
- командный file input импорта JSON;
- основные динамические группы рабочего пути: задачи, шаблоны, сценарии, компоновки, количество на А4, Wizard Flow, быстрые исправления качества и редактирование на макете.

## Типы проверки

- `browser` — действие фактически выполняется в `tools/browser-smoke.html` либо профильном `tools/ui-actions-smoke.html`;
- `screenshot` — эффект проверяется печатным screenshot-сценарием;
- `contract` — selector, обработчик и обязательные связи защищены профильным validator;
- `manual` — действие имеет внешний эффект, который нельзя честно завершить в headless Chrome, например системная печать;
- `planned` — обработчик существует, но полный пользовательский browser-сценарий ещё должен быть добавлен по issue #66.

`planned` не означает, что кнопка признана сломанной. Это видимый долг тестирования, который нельзя скрывать формулировкой «обработчик найден».

## Smoke действий сохранения

Browser job GitHub Actions последовательно запускает отдельными шагами:

```text
npm run smoke:browser
npm run smoke:ui-actions
```

Второй сценарий работает в отдельном временном профиле Chrome и проверяет реальными нажатиями:

- сохранение и загрузку профиля СПН;
- сохранение и загрузку именованного макета;
- сохранение и загрузку последнего макета;
- восстановление нескольких полей, а не только появление сообщения;
- фактическую запись в соответствующие ключи `localStorage`.

Системная печать и реальные пользовательские файлы этот сценарий не затрагивает. При ошибке он пишет в общий `browser-smoke-failure.log` последнее действие и компактный снимок полей.

## Что проверяет validator

`tools/validate-ui-actions.mjs` останавливает общий `npm run validate`, если:

- JSON не читается или нарушена схема;
- повторяется `id` либо selector;
- отсутствуют owner, ownerToken, ожидаемый эффект, риск или тип проверки;
- файл-владелец не существует или не содержит заявленный token;
- файл проверки не существует или не содержит marker;
- статическая кнопка из `index.html` не внесена в реестр;
- реестр ссылается на несуществующую статическую кнопку;
- командный `#uploadFile` потерян либо перестал быть `type=file`;
- отсутствует одна из обязательных динамических групп рабочего пути;
- package scripts или browser job перестали запускать основной и профильный smoke в установленном порядке.

## Правило сопровождения

При добавлении новой статической кнопки в `index.html` её необходимо одновременно внести в `data/ui-actions.json`. Иначе CI завершится ошибкой.

Для динамического действия следует использовать устойчивый selector: `id` или отдельный `data-*` атрибут. Селектор по случайному тексту, позиции элемента или оформительскому классу не считается стабильным контрактом.

После добавления browser smoke тип проверки нужно изменить с `planned` на `browser` и указать marker успешного сценария. Само наличие кнопки или строки `onclick` не считается браузерным подтверждением.

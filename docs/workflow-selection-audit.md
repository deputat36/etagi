# Аудит цепочки выбора рабочего пути СПН

Проверенный `main`:

```text
bb24feb2d71f11a2c0f8ee35379530b5c418200e
```

Область аудита:

```text
рабочая ситуация → задача → сценарий → шаблон
```

Machine-readable карта:

```text

```

## Текущее поведение

### 1. Рабочая ситуация

Владелец: `assets/js/spnWizard.js`.

Выбор ситуации одновременно:

- выделяет кнопку ситуации;
- программно нажимает кнопку задачи;
- подставляет поисковый запрос;
- меняет количество макетов на A4;
- применяет режим компоновки;
- выводит рекомендацию и метрику.

Изменения выполняются не одной операцией. Сначала происходит программный клик по задаче, затем через таймер 90 мс применяются поиск и рекомендуемые настройки.

### 2. Задача

Владелец: `assets/js/app.js`.

Выбор задачи:

- меняет `state.goal`;
- сбрасывает сценарий в `all`;
- автоматически применяет первый подходящий шаблон;
- полностью перерисовывает интерфейс.

Поэтому выбор рабочей ситуации косвенно заменяет текущий макет ещё до явного выбора карточки шаблона.

### 3. Сценарий

Владелец: `assets/js/app.js`.

Сценарий не меняет макет. Он дополнительно фильтрует уже ограниченный задачей, поиском и плотностью список шаблонов.

После выбора рабочей ситуации сценарий остаётся `Все`, хотя поисковый запрос ситуации уже выполняет похожее смысловое сужение.

### 4. Шаблон

Владелец: `assets/js/app.js`.

Явный выбор карточки:

- заменяет полное состояние макета данными шаблона;
- сохраняет профиль СПН и загруженные фото;
- синхронизирует задачу с `template.goal`;
- сохраняет `templateId`.

Активный сценарий при этом отдельно не пересчитывается.

## Подтверждённые проблемы

### WF-01. Автоматический первый шаблон

Задача должна объяснять тип результата и фильтровать библиотеку. Сейчас она одновременно заменяет макет первым найденным шаблоном. Пользователь может не заметить, что выбор текста уже произошёл.

### WF-02. Две фазы ситуации

Программный клик задачи создаёт промежуточное состояние. Через 90 мс отдельно меняются поиск, количество и компоновка. Это создаёт лишние рендеры и делает цепочку сложнее для сопровождения.

### WF-03. Поиск и сценарий

Поисковая фраза ситуации и сценарный фильтр работают поверх одной библиотеки. В простом рабочем пути это выглядит как повторный выбор похожего смысла.

### WF-04. Поиск засчитывается как ситуация

Маршрут `0/5` считает шаг `Ситуация` выполненным при любом непустом `templateSearch`, даже без активной офисной ситуации.

### WF-05. Задача и сценарий после шаблона

Шаблон синхронизирует `goal`, но не определяет новое состояние сценария. Необходимо выбрать единый источник истины.

## Рекомендуемый следующий PR

Приоритет: WF-01 и WF-02.

Безопасное направление:

1. Выбор ситуации одним обновлением задаёт goal, поиск и рекомендуемые параметры.
2. Произвольный первый шаблон не применяется автоматически.
3. Если ситуация имеет явно рекомендованный шаблон, он указывается отдельным ID и показывается пользователю как рекомендация.
4. В режимах `Новичок` и `Быстро` сценарий показывается как дополнительный фильтр только при необходимости.
5. Текущий макет не заменяется до явного выбора шаблона.

Не входят в этот аудит:

- изменение рекламных текстов;
- менеджерские решения по чувствительным шаблонам;
- печатные форматы и CSS;
- выпуск 3.86.0.

- явное применение карточки шаблона;
- засчитывание ручного поиска как шага `Ситуация` после сброса.
